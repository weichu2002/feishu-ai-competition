# FlowMate 正式架构

本文档是 FlowMate 的正式架构说明，用于统一后续设计、开发、排查和交接。

适用范围：
- 产品方案确认
- 模块职责划分
- 新功能接入
- 排查“这一步到底该谁做”

如果后续讨论和本文档冲突，以本文档为准；需要变更架构时，应先更新本文档。

---

## 1. 产品定位

FlowMate 是一个“会议驱动的个人承诺闭环 Agent”。

它不是通用秘书，也不是全量聊天理解器。它只聚焦一件事：

- 识别用户自己在飞书里许下的承诺
- 把承诺变成可追踪的结构化记录
- 在飞书里形成闭环

FlowMate 的核心目标不是“理解所有消息”，而是“别让用户忘掉自己答应过什么”。

---

## 2. 核心原则

### 2.1 唯一主账本

FlowMate 的正式主账本是飞书多维表格 `承诺账本`。

不再把本地 JSON 账本视为正式账本。

本地文件可以保留，但只能用于：
- 临时缓存
- 调试
- 最近一次抽取结果存档

不能再作为“系统真实状态”的唯一来源。

### 2.2 通道、业务、执行分层

FlowMate 按三层划分：

- `openclaw-lark`
  - 负责飞书消息入口和出口
  - 负责监听飞书消息、回复飞书消息、发送卡片
- `FlowMate`
  - 负责承诺识别和业务编排
  - 负责规则筛选、上下文抽取、字段标准化、去重判断、提醒策略
- `lark-cli`
  - 负责对飞书对象执行真实操作
  - 负责写多维表格、创建任务、创建日历、查询会议纪要等

### 2.2.1 唯一飞书通道约束

以下约束是 FlowMate 的硬边界，后续必须遵循：

- `openclaw-lark` 是 FlowMate 唯一正式飞书通道
- 用户在飞书里与 Bot 的私聊、群里 `@Bot`、卡片交互、消息回执、提醒推送，都只能通过 `openclaw-lark` 进出
- `openclaw-main/extensions/feishu` 及其 dist/runtime 产物，不得作为 FlowMate 的正式飞书消息通道
- 排障时可以参考或对照内置 Feishu 实现，但不得把修复落在内置 Feishu 通道上替代 `openclaw-lark`
- 任何新功能如果会影响“我能不能直接在飞书里和 Bot 对话”，默认归 `openclaw-lark`

### 2.3 先筛选，再抽取

FlowMate 不对每条消息直接跑模型。

正式链路必须分两层：

- 第一层：规则筛选
- 第二层：上下文抽取

只有命中规则的消息，才允许进入抽取阶段。

### 2.4 先面向“用户本人承诺”

第一阶段自动识别范围只覆盖：

- 用户本人发出的消息

暂不默认自动监听和处理所有群成员的承诺。

原因：
- 更轻
- 更稳
- 噪音更少
- 更符合“个人承诺闭环”的产品定位

## 3. 三个外部项目的正式分工

### 3.1 openclaw

`openclaw` 是 Agent 运行时和 Gateway。

负责：
- 跑 Gateway
- 管理 session / workspace / skill / tool
- 承接来自飞书通道的消息
- 把消息交给 Agent 处理

不负责：
- 承诺识别逻辑
- 飞书业务对象具体写入
- FlowMate 的正式飞书消息通道实现

### 3.2 openclaw-lark

`openclaw-lark` 是 OpenClaw 的飞书插件。

负责：
- 监听飞书消息事件
- 把消息送入 OpenClaw
- 把结果回复回飞书
- 提供飞书消息与卡片相关通道能力

在 FlowMate 里的正式职责：
- 唯一正式飞书 Bot 通道层
- 唯一用户对话入口 / 出口
- 不承担正式飞书对象执行主链路

也就是说，FlowMate 不以 `openclaw-lark` 的 `feishu_*` 工具作为主执行层，而是把它当成：
- 消息入口
- 消息出口

### 3.3 lark-cli

`lark-cli` 是飞书官方 CLI。

负责：
- 配置应用
- 登录授权
- 调用飞书对象能力
- 写多维表格
- 管理任务
- 操作日历
- 查询会议纪要等

在 FlowMate 里的正式职责：
- 唯一飞书对象读写执行层
- 用户身份消息搜索 / 历史读取执行层

也就是说，只要 FlowMate 需要真正操作飞书对象，应统一走 `lark-cli`，而不是一部分走 `openclaw-lark`，另一部分走 CLI。

同时，`lark-cli` 不能替代飞书 Bot 通道本身；它负责执行，不负责承接用户在飞书里的正式对话入口。

## 4. 正式目标链路

FlowMate 的正式自动识别链路如下：

`用户本人发消息`
-> `openclaw-lark 监听到消息事件`
-> `FlowMate 规则筛选`
-> `FlowMate 取局部上下文`
-> `FlowMate 抽取结构化承诺`
-> `FlowMate 决定后续动作`
-> `lark-cli 写入承诺账本`
-> `lark-cli 创建任务 / 创建日历`
-> `openclaw-lark 回执结果 / 发送提醒`

### 4.1 正式对话链路

凡是用户直接在飞书里和 FlowMate 对话，正式链路固定为：

`用户在飞书里发消息`
-> `openclaw-lark 接收消息事件`
-> `OpenClaw 路由给 FlowMate`
-> `FlowMate 做意图理解 / 承诺识别 / 目标定位`
-> `如需操作飞书对象则调用 lark-cli`
-> `FlowMate 组织回复`
-> `openclaw-lark 回消息 / 发卡片`

### 4.2 正式扫描链路

凡是“自动记录我发出的消息”这类扫描型能力，正式链路固定为：

`定时器或手动触发`
-> `FlowMate 调用 lark-cli 读取用户身份消息`
-> `FlowMate 做新增判断 / 规则筛选 / 上下文抽取`
-> `lark-cli 写承诺账本 / 任务 / 日历`
-> `如需通知则由 openclaw-lark 回执`

## 5. 正式能力范围

### 5.1 自动监听范围

第一阶段仅监听：

- 用户本人发出的飞书消息

推荐场景：
- 用户与 Bot 的私聊
- 指定群聊
- 指定会话

不建议第一阶段就默认监听：
- 所有群成员消息
- 所有历史消息
- 所有文档评论

### 5.2 识别对象

第一阶段只自动识别：

- 用户本人许下的承诺

例如：
- 我明天下午前把首页原型发出来
- 我下周补完埋点方案
- 我先去确认权限
- 我会后同步结论

### 5.3 非目标

以下内容不属于第一阶段自动识别目标：

- 所有团队成员的承诺自动抓取
- 全量聊天语义理解
- 全企业级任务治理
- 无边界群聊监控

---

## 6. 两阶段识别策略

### 6.1 第一层：规则筛选

目标：
- 快速判断一条消息是否疑似承诺
- 尽可能便宜地过滤掉无关消息

规则命中示例：
- `我来`
- `我负责`
- `我先去`
- `我明天`
- `我下午`
- `我会后`
- `下周我`
- `月底前`
- `发出来`
- `补完`
- `联系`
- `确认`
- `同步`
- `整理`
- `提交`
- `跟进`

规则过滤示例：
- `收到`
- `好的`
- `ok`
- `赞同`
- `我觉得`
- `可以考虑`
- 单纯汇报已完成且无后续动作

这一层要求：
- 不调用模型
- 不消耗额外 token
- 响应足够快

### 6.2 第二层：上下文抽取

只有第一层命中的消息，才进入第二层。

目标：
- 判断这条消息是否真的是承诺
- 抽取结构化字段
- 补齐上下文引用

这一层使用：
- 局部上下文
- 结构化抽取逻辑
- 必要时调用模型

推荐上下文窗口：
- 当前消息
- 前面 3 到 6 条相关消息
- 或最近 2 到 5 分钟同线程消息

禁止：
- 直接把整个群聊历史喂给模型
- 每条消息都跑全量上下文抽取

## 7. 承诺结构化标准

FlowMate 抽取出的承诺，正式字段至少包括：

- `id`
- `title`
- `owner`
- `ownerOpenId`
- `deadlineText`
- `deadline`
- `priority`
- `status`
- `sourceType`
- `sourceTitle`
- `sourceLink`
- `evidence`
- `confidence`
- `nextAction`
- `riskReason`
- `feishuTaskId`
- `bitableRecordId`
- `createdAt`
- `updatedAt`

其中几个关键要求：

### 7.1 证据链是必需的

每条承诺必须保留原文证据：
- 原句 quote
- 发言人
- 时间
- 来源

### 7.2 时间要双存

必须同时保存：
- `deadlineText`
  - 原始人话，例如“明天下午前”
- `deadline`
  - 标准时间，例如 ISO 时间

### 7.3 已完成不进入待跟踪账本

像：
- 我已经做完了
- 我已经改完了

如果没有后续动作，不应进入待跟踪承诺。

### 7.4 风险承诺要保留风险标记

像：
- 我先去确认权限
- 这块可能会卡一下

应允许作为承诺进入，但需要：
- `status = blocked / pending`
- `riskReason` 写清楚

## 8. 正式主账本：飞书“承诺账本”

FlowMate 的正式主账本是飞书里的多维表格 `承诺账本`。

它承担：
- 承诺登记
- 状态追踪
- 提醒扫描基础
- 复盘统计基础

正式要求：
- 所有自动识别出的承诺，先写入 `承诺账本`
- 后续任务、日历、提醒，都以这张表为准

不允许再以本地 JSON 当唯一真相来源。

---

## 9. 任务与日历的正式分工

### 9.1 任务

飞书任务用于：
- 跟踪执行动作
- 查看完成状态
- 推动真正完成

适合创建任务的承诺：
- 明确动作
- 需要执行跟踪
- 有负责人

例如：
- 补完埋点方案
- 联系设计确认图标规范

### 9.2 日历

飞书日历用于：
- 时间提醒
- 节点触达
- 明确时间点的到期提醒

适合创建日历的承诺：
- 有明确时间
- 更偏“到点提醒”

例如：
- 明天下午 3 点前发原型
- 周三上午同步评审结论

### 9.3 是否同时创建

不是所有承诺都需要同时建任务和日历。

建议规则：
- 明确动作但时间模糊：优先任务
- 时间明确但动作偏提醒：优先日历
- 重要事项：任务 + 日历都建

---

## 10. 定时提醒与复盘

提醒和复盘都以飞书 `承诺账本` 为数据源。

### 10.1 定时提醒

定时扫描：
- 今天到期
- 明天到期
- 已逾期
- 长时间无更新

提醒出口：
- 飞书 Bot 私聊
- 飞书卡片消息

### 10.2 晚间复盘

基于承诺账本生成：
- 今日新增承诺
- 今日完成承诺
- 明日到期承诺
- 风险项

---

## 11. 模块正式职责

### 11.1 `openclaw-lark`

保留职责：
- 接收飞书消息
- 发送飞书回复
- 发送卡片
- 作为自动监听入口

不再承担：
- 正式主账本写入
- 正式任务主链路执行
- 正式日历主链路执行

### 11.2 `FlowMate`

正式负责：
- 规则筛选
- 上下文抽取
- 承诺字段标准化
- 时间标准化
- 风险识别
- 去重规则
- 决策：写表 / 建任务 / 建日历 / 只记录
- 提醒和复盘逻辑

### 11.3 `lark-cli`

正式负责：
- 写 `承诺账本`
- 更新 `承诺账本`
- 创建任务
- 更新任务
- 创建日历提醒
- 查询会议纪要、任务、表格等飞书对象

## 12. 现有代码的架构调整方向

### 12.1 保留

- `src/model-client.js`
  - 保留为抽取层
- `src/types.js`
  - 保留为结构化标准
- `src/cards.js`
  - 保留为提醒/复盘卡片层
- `src/lark-cli.js`
  - 保留为 CLI 适配层
- `src/feishu-write.js`
  - 保留，但定位改成“飞书执行适配层”

### 12.2 需要降级

- `src/ledger.js`
  - 从“正式账本层”降级为“本地缓存/调试工具”

### 12.3 需要重构

- `scripts/assistant-entry.js`
  - 不能再围绕“本地账本 -> 再同步”
  - 需要改成“抽取 -> 直接写飞书”

### 12.4 需要明确废弃的设计

以下设计应视为旧方案：
- 本地 JSON 作为主账本
- Bot 和脚本分别维护两套状态
- 飞书对象一部分走 `openclaw-lark`，一部分走 `lark-cli` 的混合主链路
- 由 `openclaw-main/extensions/feishu` 承接 FlowMate 的正式飞书会话
- 内置 Feishu 通道和 `openclaw-lark` 双通道同时接管同一个 FlowMate Bot

### 12.5 必须遵守的实现红线

以后新增功能时，必须先套这张表：

- 飞书里用户和 Bot 的直接对话：只能落到 `openclaw-lark`
- 用户身份消息扫描、飞书对象读写：只能落到 `lark-cli`
- 业务判断、承诺抽取、上下文组装、目标解析：只能落到 `FlowMate`
- 不允许为了“先救活”而把正式修复落到 `openclaw-main/extensions/feishu`
- 不允许再引入第二条飞书主链路

## 13. 正式数据流

### 13.1 自动识别链路

1. 用户本人发送飞书消息
2. `openclaw-lark` 收到消息事件
3. FlowMate 规则筛选
4. 命中后取局部上下文
5. FlowMate 抽取承诺
6. 写入飞书 `承诺账本`
7. 根据规则创建任务 / 日历
8. Bot 返回处理结果

### 13.2 主动查询链路

1. 用户询问“我有哪些承诺”
2. FlowMate 读取飞书 `承诺账本`
3. 生成结构化摘要
4. Bot 回消息或卡片

### 13.3 定时提醒链路

1. 定时任务触发
2. FlowMate 读取飞书 `承诺账本`
3. 识别临期/逾期/风险项
4. 生成提醒
5. Bot 发私聊/卡片

---

## 14. 第一阶段实施边界

第一阶段只实现：

- 监听用户本人消息
- 规则筛选
- 局部上下文抽取
- 直接写飞书 `承诺账本`
- 创建任务或日历
- 简单定时提醒

第一阶段不做：

- 自动抓取所有成员承诺
- 大规模历史聊天回溯
- 复杂组织级权限治理
- 多账本并存

## 14.1 团队级扩展实施边界

团队级能力是 V2 扩展，不得破坏 V1 个人承诺闭环。

当前团队级正式实现采用：

`显式固定来源`
-> `FlowMate 团队扫描器`
-> `FlowMate 承诺 / Todo 抽取`
-> `lark-cli 写入团队重点事项推进总表`
-> `lark-cli 创建 / 更新任务和日程`
-> `openclaw-lark 负责需要回到飞书会话里的回复`

团队级实现红线：

- 团队扫描不得默认扫描所有群聊、所有成员、所有历史消息
- 团队扫描只能处理显式配置的固定来源
- 团队总表使用独立多维表格：`FlowMate_团队重点事项推进总表`
- 团队状态文件和个人状态文件必须隔离
- 团队最近操作不得覆盖个人最近操作
- 团队扫描必须跳过 Bot 回执和系统消息
- 团队级对象读写仍统一走 `lark-cli`
- 团队级飞书消息入口仍统一走 `openclaw-lark`

当前团队级脚本入口：

- `npm run team:status`
- `npm run team:ensure`
- `npm run team:scan`
- `npm run team:warn`
- `npm run team:sync-status`
- `npm run watch:team`

需要新增团队来源时，必须先写入团队配置，不允许临时在个人 watcher 里硬编码群聊。

## 15. 架构判断的速查表

当后续开发再出现“这一步到底该谁做”时，按下面判断：

### 问题：这是飞书消息收发吗？

是：
- `openclaw-lark`

### 问题：这是 FlowMate 在飞书里的正式 Bot 对话入口吗？

是：
- `openclaw-lark`

不是：
- `openclaw-main/extensions/feishu`

### 问题：这是承诺业务判断吗？

是：
- `FlowMate`

### 问题：这是飞书对象操作吗？

是：
- `lark-cli`

### 问题：这是用户身份消息搜索、增量扫描、历史读取吗？

是：
- `lark-cli`

### 问题：这是正式数据来源吗？

是：
- 飞书 `承诺账本`

### 问题：这是临时缓存或调试数据吗？

是：
- 本地文件

## 16. 最终定案

FlowMate 的正式架构定案如下：

- `openclaw-lark`
  - 唯一正式飞书消息通道层
- `FlowMate`
  - 承诺识别与业务编排层
- `lark-cli`
  - 飞书对象执行层
- `openclaw-main/extensions/feishu`
  - 不纳入 FlowMate 正式飞书链路
- 飞书 `承诺账本`
  - 唯一正式主账本
- 飞书任务 / 飞书日历
  - 承诺执行与提醒对象层

正式主链路：

`用户本人消息`
-> `openclaw-lark 监听`
-> `FlowMate 规则筛选`
-> `FlowMate 局部上下文抽取`
-> `lark-cli 写承诺账本`
-> `lark-cli 创建任务 / 日历`
-> `openclaw-lark 回复与提醒`

正式对话链路：

`飞书用户对 Bot 发消息`
-> `openclaw-lark`
