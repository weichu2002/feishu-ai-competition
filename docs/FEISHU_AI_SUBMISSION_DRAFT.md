#### 3）项目亮点介绍

FlowMate 的核心亮点是把“知识获取”和“执行闭环”连接起来。

传统知识助手通常停留在问答层面，传统待办工具则依赖用户手动创建任务。FlowMate 选择从真实办公场景里最容易丢失的信息入手：聊天、文档、会议妙记中的承诺和 Action Items。

项目亮点：

- 不只是聊天 Bot，而是能执行真实 Feishu 操作的 Agent。
- 支持从消息、文档、飞书妙记中提取承诺和待办。
- 使用 Feishu 多维表格作为正式账本，沉淀结构化项目状态。
- 自动创建任务和日历提醒，形成从消息到执行的闭环。
- 支持用户在 Feishu 里直接控制监听状态，不依赖终端。
- 支持撤销、延期、完成等已有承诺操作，并同步更新或清理表格、任务和日程。
- 支持团队固定来源扫描和团队推进总表。
- 支持基于证据的知识问答，回答中保留来源和原文上下文，降低幻觉。
- 提供本地网页控制台，方便演示和日常使用。
- 提供 live regression 测试矩阵，减少“修一个功能坏另一个功能”的风险。

#### 4）AI 亮点介绍

FlowMate 的 AI 使用不是简单调用模型聊天，而是将模型嵌入到一个可验证、可回滚、可落地的工作流中。

AI 工程化亮点：

- 规则预筛选 + 模型抽取：不是每条消息都交给模型，而是先用规则过滤可能的承诺，减少噪声和误触发。
- 上下文增强：识别承诺时拉取上下文、来源信息、原文证据，避免只看单句造成误判。
- 结构化抽取：模型输出被约束为承诺字段，包括标题、负责人、截止时间、状态、置信度、证据。
- 证据链设计：每条承诺都保留原文 quote、来源标题、来源类型、上下文摘要。
- 意图路由：区分普通问答、监听控制、承诺修改、承诺撤销、团队知识问答等不同路径。
- 可执行闭环：AI 抽取结果不是停留在文本，而是进入多维表格、任务、日历等真实 Feishu 对象。
- 回归验证：新增功能后通过脚本验证通信层、模型问答、个人扫描、账本、任务、日历、团队来源等链路。

人和 AI 的分工：

- 人负责判断产品方向、真实使用体验、比赛场景和演示重点。
- AI 辅助完成代码实现、架构梳理、错误定位、接口调试、测试脚本编写和文档整理。
- FlowMate 作为运行时 Agent，负责在办公场景中持续扫描、提取、写入、提醒和问答。

模型选型思路：

- 普通问答使用轻量模型，保证响应速度和成本。
- 承诺抽取更重视结构化稳定性，通过规则预筛选和后处理降低模型不稳定性。
- 对需要高准确性的链路保留证据字段和回归测试，而不是完全依赖模型自由生成。

引入 AI 后的工作流改变：

- 过去：开会或聊天后，用户需要手动整理待办、创建任务、设置日历、同步进展。
- 现在：用户正常聊天和开会，FlowMate 自动识别承诺、创建任务、设置提醒，并在需要时基于证据回答“谁要做什么、进展如何、风险在哪里”。

## 三、其他信息 - 自由发挥区

### 未来展望：从执行助手到 AI-Native Office System

我认为 FlowMate 现在已经能证明“飞书生态 + OpenClaw + CLI + AI Agent”可以落地解决真实办公问题，但它还没有到真正颠覆办公范式的程度。

现在的 FlowMate 主要完成的是“把散落的信息变成可执行事项”。这已经比传统待办工具更进一步，但未来更有想象力的方向是：让办公系统不再围绕“人手动维护表格和任务”，而是围绕“自然协作产生结构化事实”。

未来可以继续深化为以下几个方向：

#### 1. 从 Todo 提取升级为组织记忆

未来 FlowMate 不应只提取待办，而应该沉淀一个持续增长的项目记忆图谱：

- 谁承诺了什么。
- 为什么要做。
- 来源证据是什么。
- 依赖谁。
- 当前状态如何。
- 哪些决策影响了后续计划。
- 哪些风险反复出现。

这样团队不再依赖“翻聊天记录”和“问老人”，而是拥有一个可追溯、可问答、可执行的组织记忆层。

#### 2. 从被动 dashboard 升级为主动管理 Copilot

目前 dashboard 主要用于展示指标。下一步可以让 FlowMate 主动生成管理动作：

- 每周自动推送团队推进摘要。
- 自动识别逾期风险和阻塞原因。
- 自动提醒负责人补充状态。
- 对识别不到负责人的事项发起确认。
- 对长期未更新的事项自动追问。

这会让 AI 不只是整理信息，而是参与团队节奏管理。

#### 3. 从单人闭环升级为多人协作闭环

当前 Demo 受限于真实成员环境，主要以单用户和模拟负责人验证为主。未来需要完整支持：

- 多成员身份映射。
- 负责人识别。
- 自动分派任务。
- 私聊通知责任人。
- 管理员确认流。
- 多人状态同步。
- 跨来源去重。

这一步完成后，FlowMate 才会真正成为团队级推进中枢。

#### 4. 从固定来源扫描升级为场景化知识分发

比赛题强调“主动分发与推送”。未来 FlowMate 可以在更合适的时间点推送知识：

- 会前：根据日历和参会人推送背景文档、历史决策和风险。
- 会后：自动整理 Action Items 并分发给负责人。
- 截止前：提醒负责人并附上相关上下文。
- 出现阻塞时：检索历史类似问题和解决方案。
- 周报时：生成高密度团队进展和风险洞察。

这会让知识从“需要人去搜”变成“在正确时刻主动出现”。

#### 5. 从办公自动化升级为新的工作入口

传统办公软件让人围绕工具转：开会、写文档、建任务、填表格、同步进度。

FlowMate 的长期目标是反过来：人只需要自然协作，AI Agent 自动把协作过程转化成结构化、可追踪、可验证、可执行的工作系统。

这也是我认为这个方向最有价值的地方：它不是简单减少几次点击，而是尝试改变信息流转和执行管理的入口。

### 当前不足与改进计划

当前版本仍有不足：

- 多人真实团队验证还不够充分。
- 可视化 dashboard 还可以更美观、更管理化。
- 任务/日历/多维表格状态回写仍以周期同步为主，事件驱动能力还可以加强。
- 文档评论、邮件、更多企业知识库来源还可以继续接入。
- 对复杂语义承诺、多人责任拆分、模糊截止时间的识别还需要更多评测数据。

后续计划：

- 补充真实多人协作测试。
- 建立更系统的评测集，衡量识别准确率、负责人准确率、去重率、证据引用准确率和幻觉率。
- 将 dashboard 升级为更完整的管理驾驶舱。
- 加强状态事件回写，让任务完成、日程变更、表格状态变化能更实时同步。
- 将 FlowMate 从“承诺闭环 Agent”扩展为“团队知识与执行操作系统”。
